# Change Requirements Discovery

Your goal is to extract requirements from Jira.

Use the available tools to find requirements for current PR. Prefer deterministic sources over guesses:

- First inspect the PR metadata/description. If it contains an explicit requirements link, use that link and do not search nearby issue numbers.
- Use `gh_pr` to retrieve GitHub PR metadata, including title, branch names, URL, and description/body. If no PR number is known, call it without a PR number to retrieve the current branch PR.
- Look in branch name or PR description, for something that looks like Jira issue number which consists of 2 or more letters, dash and a number, for example: PP-1234, PR-12345, DO-125, etc., this could be a part of a code itself in PR title, a link in PR description, branches usually have the following format feature/PP-1234-add-useful-feature, where PP-1234 is a Jira issue key.  
- Use Jira MCP to fetch a title and description of this Jira

When you have the best requirements available, call `set_requirements` with the complete requirements text exactly as they should constrain the review.

Do not invent requirements. Do not brute-force issue numbers. If no requirements can be found, call `set_requirements` with a concise statement explaining that no external requirements were found and what evidence was checked.

Finish with a short summary of what you set and where it came from.
